## Context

当前开发流程中，测试一个文件的多个风格变体需要串行执行：每次运行一个风格，等待完成，查看结果，再启动下一个。这个过程存在三大痛点：
1. **效率低**：多风格需要多次等待，总等待时间 = 单变体时间 × 变体数量
2. **不可见**：运行时无法看到中间过程，只能等待最终结果
3. **交互繁琐**：每次都要重新输入相同的参数（文件路径），只有风格不同

本设计利用 tmux 的 pane 分割能力实现并行执行，利用 kitty 的图像展示能力实现快速预览。

## Goals / Non-Goals

**Goals:**
- 支持一次启动多个变体并行运行
- 实时显示每个变体的输出日志
- 运行结束后自动展示结果图片
- 提供交互式 CLI 降低使用门槛
- 降级兼容：无 kitty 时用 Python PIL 展示

**Non-Goals:**
- 不改变现有单变体 CLI 的行为
- 不支持分布式/远程执行（仅限本机 tmux）
- 不提供 GUI 界面（纯终端方案）

## Decisions

### 1. 使用 tmux 作为窗口管理核心
**决策**：使用 tmux + tmuxifier 管理多 pane 布局

**理由**：
- tmux 是本机终端会话的标准工具，稳定可靠
- tmuxifier 支持预设布局，可快速恢复环境
- pane 之间天然隔离，每个变体独立运行
- 支持会话保持，中断后可恢复查看

**替代方案考虑**：
- GNU Screen：功能类似但社区活跃度低，kitty 集成不如 tmux
- 纯 Python 多进程 + 日志轮询：复杂度高，失去终端交互能力
- 桌面 GUI：开发成本高，违背终端工作流

### 2. 使用 kitty 的 kittens 协议展示图片
**决策**：优先使用 `kitten icat`，降级为 Python PIL

**理由**：
- `kitten icat` 是 kitty 原生图片协议，展示速度快、质量好
- 降级方案保证在非 kitty 终端仍可运行
- Python PIL + base64 可在 kitty 中回退展示

**替代方案考虑**：
- 纯 ASCII art：质量太差，不适合设计评审
- 外部图片查看器：打断终端工作流
- 仅支持 kitty：降低兼容性

### 3. Python 封装脚本
**决策**：使用 Python 而非 Bash 封装

**理由**：
- Python 已有项目依赖（见 `src/narrio/`）
- 更好的错误处理、参数解析、交互逻辑
- 可复用现有 highlight_service.py 等模块
- 更易维护单元测试

**替代方案考虑**：
- Bash：更轻量但错误处理复杂，交互逻辑难写
- 混合方案（Bash 启动 + Python 核心）：增加复杂度

### 4. 双窗口布局
**决策**：一个多 pane 窗口（运行区）+ 一个单 pane 窗口（汇总区）

**理由**：
- 运行区：每个 pane 独立显示变体的实时输出
- 汇总区：聚合所有变体的完成状态和结果路径
- 符合"并行执行 → 统一评审"的工作流

**布局示例**：
```
[运行区 - 多 pane]          [汇总区 - 单 pane]
┌─────────┬─────────┐      
│ style1  │ style2  │  ←  汇总日志
├─────────┼─────────┤      - style1: done
│ style3  │ style4  │  - style2: running
└─────────┴─────────┘  - style3: pending
```

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 用户无 tmux 安装 | 启动前检查并给出安装指引 |
| 用户无 kitty | 降级为 Python PIL 展示 |
| tmux 会话冲突（同名） | 使用唯一命名：`narrio-tune-<timestamp>` |
| pane 输出混乱 | 每个 pane 运行独立子 shell，隔离环境变量 |
| 图片过大无法展示 | 限制最大尺寸，提供本地路径 fallback |

## Migration Plan

1. **开发阶段**：
   - 创建 `src/narrio/tune.py` 脚本
   - 添加 `narrio tune` CLI 入口（若项目有 CLI 框架）
   - 编写 tmuxifier 配置文件（可选）

2. **测试阶段**：
   - 手动测试：多风格并行运行
   - 回归测试：确保单变体 CLI 不受影响

3. **部署**：
   - 无需数据库迁移或配置变更
   - 纯代码新增，不影响现有功能

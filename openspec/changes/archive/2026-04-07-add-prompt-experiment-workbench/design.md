## Context

Narrio 当前以 Python 脚本和文件系统串联内容生成流程，已经验证了从文章或播客输入到 chunk、editorial、图片输出的基本可行性。但现状更适合顺序执行单次任务，不适合开发者高频迭代 prompt：实验目录不稳定、中间产物追踪不统一、阶段重跑依赖手工拼接路径，CLI 也缺少面向开发者的友好交互。

本次实现需要优先服务开发者工作流，而不是直接建设最终产品形态。与此同时，终端交互只是临时入口，未来 PWA 仍需复用同一套 workflow 和 service 边界，因此核心执行逻辑不能绑定在 `fzf` 或 CLI 分支中。

## Goals / Non-Goals

**Goals:**
- 引入本地 gitignored workbench 目录，为每个输入组合和运行实例提供稳定的落盘位置
- 定义统一的实验运行模型，包含 `combo-id`、`run-id`、manifest、事件日志和中间产物目录
- 把现有 chunkify、stylify、render 收敛成可复用步骤，支持从 `source`、`chunk`、`editorial` 开始重跑
- 提供开发者 CLI，兼容命令式调用和交互式调用两种模式
- 让交互层只负责收集参数，执行层保持可被未来 PWA 复用

**Non-Goals:**
- 不建设 Web UI 或 PWA 页面
- 不引入数据库、任务队列或多服务架构
- 不在这一轮解决所有 prompt 版本管理和结果对比可视化问题
- 不重写现有生成逻辑本身，只做结构化收口和入口改造

## Decisions

### 决策 1：使用 `.narrio/workbench` 作为本地实验根目录

采用 gitignored 的 `.narrio/workbench/<content-type>/<combo-id>/runs/<run-id>/` 结构保存实验数据。

原因：
- 开发者需要完整保留 request、response、chunk、editorial 和 render 结果
- 实验数据天然是临时和高频产物，不适合直接进入版本库
- 文件系统对当前阶段最直接，便于手动检查和恢复

备选方案：
- 直接写入 `output/`：目录语义偏最终产物，不利于实验归档
- 引入数据库：对当前原型过重，且不利于查看中间文件

### 决策 2：把实验身份拆分为 `combo-id` 与 `run-id`

`combo-id` 表示一组输入选择，`run-id` 表示在该组合下的一次具体执行。

原因：
- 同一输入组合下需要多次试验不同 prompt 内容或参数
- 允许一个组合目录中保留多次执行，并提供 `latest` 快捷指向
- 有利于未来做按组合聚合比较，而不是只按单次 run 比较

备选方案：
- 只保留 `run-id`：难以表达同一组合下的多次试验关系
- 只保留 `combo-id`：无法区分历史执行记录

### 决策 3：将现有脚本能力收敛为步骤服务，再由 pipeline 编排

保留现有脚本中的核心生成逻辑，但将 chunkify、stylify、render 封装为可复用步骤服务。

原因：
- 阶段重跑要求可以单独调用某一步
- CLI 和未来 PWA 需要复用相同执行入口
- 可以在不重写生成逻辑的前提下降低脚本耦合

备选方案：
- 继续直接串联原脚本：实现快，但后续难以支持 resume 和统一日志
- 一次性全面重写：风险过高，不适合当前原型阶段

### 决策 4：重跑能力显式限定为 `from-source`、`from-chunk`、`from-editorial`

实验 pipeline 只支持三个起始阶段，且要求对应中间产物已存在并兼容。

原因：
- 这三个阶段正好覆盖当前工作流的主要成本点
- 能满足大多数 prompt 迭代需求
- 限定边界可以避免隐式回退和状态混乱

备选方案：
- 自动探测最近可用阶段并回退：更“智能”，但容易让结果不可解释
- 细化更多起始阶段：实现复杂度高，短期收益低

### 决策 5：CLI 采用“命令模式 + 交互模式”双轨设计

提供命令式入口承接自动化和脚本化，同时提供交互式入口承接本地实验。

原因：
- 命令模式便于验证、测试和后续前端调用适配
- 交互模式降低开发者配置实验的心智成本
- 同一套底层执行逻辑可以服务两种入口

备选方案：
- 只做交互式 CLI：体验友好，但自动化能力差
- 只做命令式 CLI：可维护，但不利于高频实验

### 决策 6：交互模式优先依赖 `fzf`，并提供降级路径

当 `fzf` 可用时使用列表选择与预览；不可用时回退到普通文本输入。

原因：
- `fzf` 是开发者终端中成熟且高效的交互工具
- 不强制新增复杂 TUI 框架
- 保留跨环境可执行性

备选方案：
- 引入完整 TUI 框架：能力更强，但当前投入过大
- 完全不做交互增强：无法满足本次开发者效率目标

## Risks / Trade-offs

- [工作台目录增长过快] → 通过 `latest` 链接、按组合归档和后续清理命令降低成本
- [在旧脚本上做收口仍保留部分重复逻辑] → 本轮接受有限重复，优先保证运行模型成型
- [`fzf` 依赖在部分环境缺失] → 提供文本模式降级，不让交互能力成为硬依赖
- [阶段重跑可能误用过期中间产物] → 在 manifest 中记录资产和 schema 摘要，缺失或不兼容时直接失败
- [先做 CLI 可能看起来偏离 PWA 目标] → 明确 CLI 只负责参数采集，workflow/service 作为长期复用边界

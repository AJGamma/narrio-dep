## Context

当前 Narrio 的 render 阶段采用严格的串行执行模式：

```
封面图 (依赖默认 ref) → 页 1(依赖封面) → 页 2(依赖封面) → ... → 页 N(依赖封面)
```

总耗时 = 封面图时间 + (N-1) × 单页时间

这种模式的问题：
- 封面图生成完成后，后续所有页面仍然顺序执行
- 每个页面都在等待前一个页面完成，即使它们都使用同一张封面图作为参考
- 对于典型 5-10 页的内容，生图阶段可能耗时 10-30 分钟

## Goals / Non-Goals

**Goals:**
- 封面图生成后，其余页面并发生成
- 保持封面图优先生成的策略（确保参考图一致）
- 支持通过 CLI 参数控制并发 worker 数量
- 保留完整的错误处理和重试机制
- 默认行为向后兼容

**Non-Goals:**
- 不改变 chunkify 和 stylify 阶段的串行执行（存在数据依赖）
- 不引入外部任务队列或分布式系统
- 不改变单页生图的超时和重试逻辑

## Decisions

### 决策 1：两阶段渲染策略

**决策**：将 render 阶段分为两个子阶段：
1. **封面图阶段**：串发生成封面图（page 0 或第一个 page）
2. **并行阶段**：使用 ThreadPoolExecutor 并发生成其余页面

**理由**：
- 所有非封面页都依赖封面图作为参考
- 封面图必须先完成，但其余页之间无依赖关系
- Python 标准库 `concurrent.futures.ThreadPoolExecutor` 足够简单可靠

**替代方案**：
- 方案 A：所有页面完全并行（不可行，违反参考图依赖）
- 方案 B：使用 asyncio 异步 IO（复杂度高，收益有限）

### 决策 2：默认最大并发度

**决策**：默认使用「剩余页数」和「上限 5」中的较小值作为并发 worker 数。新增 `--render-workers` 参数允许用户手动指定上限。

**理由**：
- 最大化并行效率，减少用户等待时间
- 5 个并发是合理上限，避免触发 API 速率限制
- 用户仍可通过参数手动调低或调高

**替代方案**：
- 固定并发数（不够灵活，页数少时浪费）
- 通过环境变量配置（不够直观）

### 决策 3：错误隔离

**决策**：单个页面生成失败不影响其他页面继续执行。

**理由**：
- 避免因单页失败导致全部重跑
- 用户可以选择性地重试失败页面

## Risks / Trade-offs

**[Risk] API 速率限制** → 并发可能触发 OpenRouter 的速率限制
- **Mitigation**: 默认并发数上限设为 5，用户可以通过参数调低

**[Risk] 内存占用增加** → 多个并发请求同时持有图片数据
- **Mitigation**: 单页图片数据量小（~1MB），并发 3-5 个影响有限

**[Trade-off] 日志可读性下降** → 并发任务的日志可能交错
- **Mitigation**: 在日志中包含页码标识，使用线程安全的日志输出

**[Risk] 临时文件管理复杂化** → 并发任务可能需要访问临时参考图
- **Mitigation**: 临时目录在所有任务完成前不删除
